同源

网络都在说如何解决跨域问题? 但是对于导致跨域的问题的同源策略却探讨不足，个人觉得很粗暴-_-!

什么是同源

同源有三个要求:

  • 同协议(http, https为不同协议)
  • 同域(store.company.com 和 company.com属于不同域)
  • 同端口 (注: 早期版本IE未将端口号加入到同源策略的组成部分之中)

store.company.com向company.com发送XHR请求时，在以下情况可以通过同源检测:

  1. 在store.company.com中设置document.domain = ‘company.com’.
  2. 需要是火狐浏览器 (是的, 在MDN上有描述, 但是在chrome下同样会报同源错误, 其它浏览器未测试)

同源策略会限制什么

  • Ajax(XHR)请求
  • font-face (存在浏览器差异，有些限制, 有些不限制, google fonts在响应头中加了Access-Control-Allow-Origin: *)
  • WebGL textures
  • 使用drawImage嵌入canvas中的图片(images drawn to a canvas using drawImage)

同源策略不限制什么

基于标签的跨域请求基本都不限制(可以这么理解: 对于静态资源一般是不限制的):

  • script
  • img
  • form post
  • object, embed, applet
  • font-face(存在浏览器差异, 有些限制, 有些不限制)
  • video, audio
  • link(各种浏览器行为有差异)
  • frame, iframe (X-Frame-Options头信息可以控制此限制)

cookie的同源策略

Cookie中的同源只关注域名，忽略协议和端口。而且子域和父域允许相互读写。

domain.com写cookie到所有子域:

document.cookie=‘parent_id=10; domain=.domain.com’;

sub.domain.com写cookie到其父域:

document.cookie=‘child_id=10; domain=domain.com’;

为什么会有同源策略

当然，这是为了网络安全, 那么没有同源策略会有哪些影响？我从网上摘录了一个场景:

Assume you are logged into Facebook and visit a malicious website in another browser tab. Without the same origin policy JavaScript on that website could do anything to your Facebook account that you are allowed to do. For example read private messages, post status updates, analyse the HTML DOM-tree after you entered your password before submitting the form.

浏览器是怎么做的

需要注意的是， 客户端发送的请求不在同源策略的限制中, 服务器端响应的数据才有跨域限制的问题, 而且本质上response已经到达本地，只不过浏览器在网络层做了一下校验, 看看服务器端是否允许此客户端访问这个页面, 不是的话浏览器会触发XMLHttpRequest’s error事件，然后拒绝把响应的数据交给请求该数据的js

如何跨站访问数据

使用CORS(Cross-Origin Resource Sharing)

这应该会成为未来解决跨域问题的主要方法, CORS标准添加了一些新的HTTP headers, 可以让客户端在已授权的情况下给远程服务器发送请求(部分验证在服务器端, 但是绝大多数在浏览器端完成), 这些相关的headers有:

Request headers:

  • Origin
  • Access-Control-Request-Method
  • Access-Control-Request-Headers

Response headers

  • Access-Control-Allow-Origin（指定哪些源可以请求到数据）
  • Access-Control-Allow-Credentials
  • Access-Control-Expose-Headers
  • Access-Control-Max-Age（指定response的缓存时间, 在这个时间内, 不需要再次发送preflight请求）
  • Access-Control-Allow-Methods（指定哪些方法可以在请求中使用）
  • Access-Control-Allow-Headers

我们只要简单的在返回的报文头中加入Access-Control-Allow-Origin就行, 比如

http.createServer(function(req, res, err) {

  // *表示所有的客户端都能跨域请求到127.0.0.1:3000的数据
  // 这里显示的设成特定的源, 如res.setHeader(‘Access-Control-Allow-Origin’, ‘http://127.0.0.1:9000’);
  res.setHeader(‘Access-Control-Allow-Origin’, ‘*’);
  res.end(‘Hello World\n’);
}).listen(3000, ‘127.0.0.1’);

至于浏览器端的实现，我们需要先区分几个概念:

simple method:: GET, HEAD, POST simple header::

A header is said to be a simple header if the header field name is an ASCII case-insensitive match for Accept, Accept-Language, or Content-Language or if it is an ASCII case-insensitive match for Content-Type and the header field value media type (excluding parameters) is an ASCII case-insensitive match for application/x-www-form-urlencoded, multipart/form-data, or text/plain. simple response header: A header is said to be a simple response header if the header field name is an ASCII case-insensitive match for one of the following:

  • Cache-Control
  • Content-Language
  • Content-Type
  • Expires
  • Last-Modified
  • Pragma

那么对non-simple method是分两个步骤执行的:

  1. 发送preflight请求
  2. 如果Access-Control-Allow-Origin包含客户端Origin，再发送正式的请求

以下是这种方法的浏览器支持情况(摘自wikipedia):

  • IE8, IE9 (部分支持, 使用XDomainRequest)
  • IE >= 10
  • Firefox >= 3
  • Webkit 支持 (Safari >= 4, Chrome >= 3)
  • Opera > 12.00
  • Opera Mini 不支持

jsonp

变相的使用<script>，地球人都知道，具体使用不赘述. 和CORS比较:

jsonp有良好的兼容, 但是只支持get请求, 并且可能导致XSS攻击，安全性不如CORS; 而CORS支持其它的请求方法, 使用更加方便, 安全性高于JSONP(允许客户端自己解析响应的数据), 因为发送的是常规的XMLHttpRequest, 所以也更容易做错误处理.

nginx反向代理

这是我个人比较喜欢的方法，兼具jsonp和CORS的优点。具体做法: 假如http://sitea.com要访问http://siteb.com/agency下的内容, 可以在nginx.conf中加入如下配置:

location /agency {
    proxy_pass   http://119.255.25.130/agency;
}
