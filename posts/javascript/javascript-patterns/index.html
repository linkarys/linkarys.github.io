Javascript模式

Apr 20, 2015 • linkarys

  • javascript
  • patterns

老生常谈, 做下总结。似乎在ES6之前, js并不算一个完备的面向对象的语言, 本文总结的可能有效期也比较有限, 但是核心思想应该是不变的。当然, 我们并不是为了模式而模式, 要知道很多模式是基于静态语言提出的, 在js中并一定有市场, 也只有在某类模式能够优雅地解决某种需求时, 才有使用的意义.

单例

使用场景

  • 计数
  • 全局配置
  • 作为模块通信的桥梁

示例

个人觉得以下是个比较好的实现, 可以省略new, 外界没法直接访问和修改instance.

var Singleton = (function() {
  var instance;

  return function single(name) {
    if (instance) return instance;

    // 处理不用new关键字, 直接调用的情况
    if (!(this instanceof single)) {
      return new single(arguments);
    }

    this.name = name;

    instance = this;
  }
}());

工厂

根据字符串指定的类型在运行时创建对象的方法

使用场景

在静态语言中用的比较多, 工厂模式有以下目标:

  • 创建相似对象时执行重复操作
  • 编译时不知道具体类型的情况下， 为工厂客户提供一种创建对象的接口(动态语言没有这个烦恼)

示例

var MobileFactory = (function() {
  var constr,
      F = new Function();

  // 使用闭包避免每次调用都要创建代理函数
  return function factory(sys) {
    constr = factory[sys];

    // 如果构造函数不存在, 抛出异常
    if (typeof factory[sys] !== 'function') {
      throw {
        name: 'Error',
        message: 'factory ' + sys + ' is not a valid construtor function'
      }
    }

    // 子类需要继承父类的方法, 这里通过constr.prototype.info判断是否已经继承过,
    // 使用代理F来提高执行效率
    if (typeof constr.prototype.info === 'undefined') {
      F.prototype = MobileFactory.prototype;
      constr.prototype = new F();
      constr.prototype.constructor = constr;
      constr.uber = factory;
    }

    return new constr();
  }
}());

MobileFactory.prototype.info = function() {
  console.log('system info: ' + this.company);
};

// 定义子类构造方法
MobileFactory.ios = function() {
  this.company = 'apple';
};

console.log(MobileFactory.ios.prototype);


var ios = MobileFactory('ios');
ios.info();                                 // system info: apple
console.log(ios instanceof MobileFactory);  // true

var android = MobileFactory('android');     // Error: factory android is not a valid construtor function

有意思的是js内部其实有个“自然工厂”, 那就是Object, 来看下面的例子:

var
    obj = Object(),
    num = Object(1),
    str = Object('str'),
    bln = Object(true);

obj.construtor === Object;                  // true
num.construtor === Number;                  // true
str.construtor === String;                  // true
bln.construtor === Boolean;                 // true

迭代器

提供一个API来遍历和操纵复杂的自定义数据结构。 在迭代模式中,通常有一个包含某种数据集合的对象。该数据可能存储在一个复杂数据结构内部,而提供一种简单的方法能够访问数据结构中的每一个元素。对象的消费者并不需要知道如何组织数据,所有需要做的就是取出单个数据进行工作。

看了以上的说明, 其实这种模式在js中并没有什么市场。尽管如此, 还是来看一下代码吧:

var agg = (function() {
  var index = 0,
      data = [1,2,3,4,5],
      length = data.length;

  return {
    next: function() {
      if (this.hasNext()) {
        return data[index++];
      }
      return null;
    },
    hasNext: function() {
      return index < length;
    },
    current: function() {
      return data[index];
    },
    rewind: function() {
      index = 0;
    }
  }
}());

装饰者

通过从预定义装饰者对象中添加功能, 从而在运行时调整对象。

目的：运行时动态添加附加功能到对象中

使用场景

个人觉得在作组件化的时候会比较有用, 因为有些组件间的差异很小, 这个时候你不会希望再建一个类似的组件, 而通过传入大量的配置参数和解决其实对使用者和开发者来说都不算友好, 同时需求的增加, 后期的维护和扩展都将是一个问题

示例

以下代码是个人理解, 不理解这么简单的例子《JavascScript模式》一书, 选择用列表或者继承来实现

var Sale = function(price) {
  this.price = price > 0 || 0;
}

// 增加联邦税
Sale.prototype.fedtax = function() {
  this.price += 20;
  return this;
};

// 增加省级税
Sale.prototype.quebec = function() {
  this.price += 10;
  return this;
};

// 如果是英雄的话, 减10　
Sale.prototype.hero = function() {
  this.price -= 10;
  return this;
};

// 测试
var sale = new Sale();
sale.fedtax().quebec().hero();

console.log(sale);  // { price: 20 }

策略

选择最佳策略以处理特定任务

没有特别之处, 只不过给一个经常用的方案取了一个名字

使用场景

表单验证

示例

var validator = {
  validate: function(data, config) {
    var checker,
        type,
        errors = [];

    for (var key in data) {
      if (data.hasOwnProperty(key)) {
        checker = config[key];
        type = validator.types[checker];

        if (!type.validate(data[key])) {
          errors.push(type.instructions);
        }
      }
    }
    return errors;
  },
  types: {}
};

// 非空检查
validator.types.nonEmpty = {
  validate: function(value) {
    return value !== '';
  },
  instructions: 'the value can\'t be empty'
};

// 是否为数字
validator.types.number = {
  validate: function(value) {
    return Object.prototype.toString.call(value) === '[object Number]';
  },
  instructions: 'the value should be number'
};

// 是否为数字和字母的组合
validator.types.alphaNum = {
  validate: function(value) {
    // 和/[a-zA-Z0-9]*/相比, 这里只要匹配到一个非字母数字就返回false, 效率更高
    return !/[^a-zA-Z0-9]/.test(value);
  },
  instructions: 'the value can only contains numbers and characters'
};

var data = {
  firstName: '',
  age: 1,
  lastName: '3*'
};

var config = {
  firstName: 'nonEmpty',
  age: 'number',
  lastName: 'alphaNum'
};

var errors = validator.validate(data, config);

console.log(errors);  // [ 'the value can\'t be empty', 'the value can only contains numbers and characters' ]

外观(facade)

通过把常用方法包装到一个新方法中，从而提供一个列为便利的API。

使用场景

同样没有特别之处, 同样只是取了一个名字。

有时候, 两个或更多的方法可能普遍的被一起调用。在这样的情况下， 创建另一个方法以包装重复的方法调用是非常有意义的。

比如stopPropagation和preventDefault常常一起使用, 我们可以创建一个外观方法从而同时调用这两个方法:

var myevent = {
  stop: function(e) {
    e.preventDefault();
    e.stopPropagation();
  }
}

代理

通过包装一个对象以控制对它的访问，其主要方法是将访问聚焦为组或公当真正必要的时候才执行访问, 从而避免了高昂的操作开销

使用场景

这个模式还是不错的, 具体应用可以是通过一个代理来合并请求: 设置一个超时时间(如50ms), 把这个时间段内的请求合并成一个以提高性能(可以同时把请求的结果缓存起来).

另一应用是延迟初始化: 如果初始化本体对象开销很大, 并且这个对象使用频率并不高。在这种情况下, 代理可以通过替换本体对象的接口来解决这个问题, 代理接收初始化请求, 但是直到该本体对象明确的将被实际使用之前, 代理不会将该请求传递给本体对象.

中介者

对象之间互相不直接“通话”, 而是仅通过一个中介者对象进行通信, 从而促进形成松散耦合

使用场景

对象之间的直接通信可能会导致不良的紧耦合, 当对象间紧密耦合的时候, 代码的维护和扩展将变得十分困难, 中介者的目的是缓解这个问题, 并促进形成松耦合.

示例

这是一个游戏的代码示例, 每次玩家play的时候都需要更新一次计分板, 但是我们并不让玩家直接更新计分板而是通过中介(mediator)来更新计分板, 以达到降低代码耦合底的需要

// 玩家
function Player(name) {
  this.name = name;
  this.points = 0;
}

Player.prototype.play = function() {
  this.points++;
  mediator.play();
};

// 计分板
var scoceboard = {
  update: function(scores) {
    var msg = '';
    for (var i in scores) {
      if (scores.hasOwnProperty(i)) {
        msg += i + ': ' + scores[i] + '\n';
      }
    }
    console.log(msg);
  }
}

// 中介
var mediator = {
  players: {},
  setup: function() {
    this.players.home = new Player('home');
    this.players.guest = new Player('guest');
  },
  play: function() {
    var self = this;
    scoceboard.update({
      'home': self.players.home.points,
      'guest': self.players.guest.points,
    });
  },
  keypress: function(e) {
    e = e || window.event;
    if (e.which === 49) {
      mediator.players.home.play();
    } else if (e.which === 48) {
      mediator.players.guest.play();
    }
  }
};

// 测试
console.log('start...');
mediator.setup();
window.onkeypress = mediator.keypress;

var timeout = setTimeout(function() {
  window.onkeypress = null;
  console.log('finished...');
}, 10000);

观察者

通过创建可观者的对象，当发生一个感兴趣的事件时可将该事件通过千给所有观察者, 从而促进形成松散耦合.

和中介相比, 观察者并不是一个对象去调用另一个对象的方法, 观察者是一个对象订阅另一个对象的特定活动并在状态变化后获得通知.

使用场景

杂志订阅等, 这里我们把中介示例中的那个游戏用观察者模式重写一下

示例

以下代码大多数源于《JavaScript》源码一书, 为让它能在控制台直接运行, 做了轻微简化。下面的代码对于观察者模式应该还算个不错的例子, 不过也不是没有改进的空间, 比如makePublish方法采用复制的方案来实现继承, 有一定性能损失, publisher 的实现不够清晰。

// Players
function Player(name, key) {
  this.name = name;
  this.points = 0;
  this.key = key;
}

Player.prototype.play = function() {
  this.points++;
  this.fire('play');
};

// 发布者应该具有的属性
var publisher = {
  // 保存定阅者列表, 在此版本代码中, 所有发布者共用这个属性
  subscribers: {
    any: []
  },

  // 事件定阅器, type: 事件名, fn: 对type事件感兴趣的方法,
  // context: fn的上下文即fn对应的this(此处说法不严谨, 上下文不仅只有this)
  on: function(type, fn, context) {
    type = type || 'any';
    fn = typeof fn === 'function' ? fn : context[fn];

    if (typeof this.subscribers[type] === 'undefined') {
      this.subscribers[type] = [];
    }

    this.subscribers[type].push({fn: fn, context: context || this});
  },

  // 移除订阅
  remove: function(type, fn, context) {
    this.visitSubscribers('unsubscribe', type, fn, context);
  },

  // 触发事件publication: 事件传递的参数
  fire: function(type, publication) {
    this.visitSubscribers('publish', type, publication);
  },

  // 对fire和remove事件的具体处理, 个人以为这部分实现不够清晰,
  // 不如直接写在remove, fire方法中
  visitSubscribers: function(action, type, arg, context) {
    var pubtype = type || 'any',
        subscribers = this.subscribers[pubtype],
        i,
        max = subscribers ? subscribers.length : 0;

    for (i = 0; i < max; i++) {
      if (action === 'publish') {
        subscribers[i].fn.call(subscribers[i].context, arg);
      } else if (subscribers[i].fn === arg & subscribers[i].context === context) {
        subscribers.splice(i, 1);
      }
    }
  }
};

// 创建发布所用的函数, 此处基于复制进行继承而非原型
// 是因为传入的参数是一个实例对象而非构造函数
function makePublisher(obj) {
  for (var i in publisher) {
    // 此处浅复制意味着publish.subscribers为所有对象共用
    if (publisher.hasOwnProperty(i)) {
      obj[i] = publisher[i];
    }
  }
}

// 计分板
var scoreboard = {
  update: function(scores) {
    for (var score in scores) {
      console.log(score, scores[score]);
    }
  }
};

// game容器
var game = {
  keys: [],
  addPlayer: function(player) {
    // 键盘按键的keycode对应其ASCII值
    var key = player.key.toString().charCodeAt(0);
    this.keys[key] = player;
  },
  // 订阅键盘事件
  handlerKey: function(e) {
    e = e || window.event;

    var player = this.keys[e.which];

    player && player.play();
  },
  // 用于订阅player的play事件
  handlerPlay: function() {
    var scores = {},
        players = this.keys;

    for (var key in players) {
      if (players.hasOwnProperty(key)) {
        scores[players[key].name] = players[key].points;
      }
    }

    this.fire('change', scores);
  }
}

// 让game和player成为事件发布者
makePublisher(game);
makePublisher(Player.prototype);

// 创建玩家
var playerA = new Player('A', 'a');
var playerB = new Player('B', 'b');

// 添加玩家
game.addPlayer(playerA);
game.addPlayer(playerB);

// 通过订阅感兴趣对象的事件建立联系
Player.prototype.on('play', 'handlerPlay', game);
game.on('change', scoreboard.update, scoreboard);

// 开始游戏
console.log('start');
window.onkeypress = function(e) {
  game.handlerKey(e);
};

// 取消订阅
setTimeout(function() {
  console.log('unsubscribe');
  game.remove('change', scoreboard.update, scoreboard);
}, 5 * 1000);

// 结束游戏
setTimeout(function() {
  console.log('finished');
  window.onkeypress = null;
}, 7 * 1000);
