移动端适配（一） — 谈谈viewport

Aug 13, 2015 • linkarys

  • css
  • viewport

viewport是我们在网站上看到的可视区域，其宽度: window.innerWidth, 高度: window.innerHeight. 我们可能通过调整浏览器窗口大小来调整viewport的大小，当页面的内容的尺寸大于viewport的尺寸时，我们需要滚动来查看整个页面(本质上是通过移动viewport来查看页面的其它内容)

很久以前，当智能手机来到这个地球上的时候，它知道世界上的网站其实还没有为它的出现作好准备。于是乎，默认情况下智能手机的浏览器都统一认为所有站点的宽度都是在980px左右，当访问一个网站时，浏览器会缩小网站以让它们适应viewport的宽度（相当于在980px宽的浏览器上查看网站，只过屏幕缩小到手机屏的大小）。就像下图这样：

当然了，这样用户体验并不好, 由此也促使了<meta name="viewport">标签诞生。

device width & height

<meta name="viewport">只对移动端浏览器有作用，并且width属性只有目标的网站的宽度小于980px, 或者为响应式布局才有需要。 我们经常可以看到这样的语句：

<meta name="viewport" content="width=device-width">

意思是将viewport的宽度设置成和设备等宽（device-width指的是设备横向css像素的数量, 比如在iphone4, iphone5中device-width为320，iphone6为375）

对于iphone6而言，由于其横向的css像素为375px, 所以以上的代码相当于

<meta name="viewport" content="width=375">

注:单位不是必需的，并且无论写了什么单位, 它始终为px，并且有些属性之前存在制约关系, 比如设置了initial-scale=1, 再设置width小于设备的横向css宽度将没有作用（下文细说）

scale

涉及四个值, 前三个的范围都是从0到10:

  • minimum-scale: 默认值: 0.25
  • maximum-scale: 默认值: 5.0
  • initial-scale: 一些情况下会被推断为1，此时浏览器会将每个css像素和viewport像素一一对应
  • user-scalable: 控制用户能否缩放， 值为yes或no

值推断和冲突

存在这么几种情况:

  1. 不添加viewport时, width默认980px（可能有浏览器差异）, initial-scale=1
  2. 当设置initial-scale为1时，width默认为device-width, height默认为device-height
  3. 当width设置小于设备横向css宽度时，页面会被放大, 即些时initial-scale的值不会为1（强制设置initial-scale为1将使刚刚设置的width无效)

w3c做了什么

meta标签最初是由apple公司提出来的，最终w3c也决定去实现标准化。但是w3c提出的方案并不是添加一个meta的标签，而是将其加入到css中，类似这样：

@viewport {
    width: device-width;
    zoom: 1;
    min-zoom: 0.5;
    max-zoom: 3;
    user-zoom: fixed;
}

看来起来确实会优雅一些，而且我们还很高兴的看到它在各浏览器中均有非常良好的实现-_-!：
